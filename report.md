# SOTA Hybrid Vision Transformer (ViT) Project: Comprehensive Report

## 1. Project Overview & Objective
The objective of this course project was to reproduce the core token-pruning idea from DynamicViT (NeurIPS 2021) on CIFAR-10 by combining **Knowledge Distillation (KD)** with **Asymmetric Token Pruning (ATP)**. The goal was to retain near-teacher accuracy while producing a "student" model that drops the large majority of its internal patch tokens, reducing compute. **Measured outcome:** the pruned student reaches **98.9%** test accuracy — only **~0.1 pt** below the same weights run without pruning, and within ~0.4 pt of the larger dense teacher-class models — while operating on only **~15% of patch tokens** in its final blocks, cutting model FLOPs by **~25%** and improving throughput by **~1.4×** at server batch sizes (see §5.3 for the full, honest efficiency breakdown, including where pruning does *not* help).

---

## 2. Model Architecture & Mechanics

### 2.1 The Core Models
The architecture is based on the Vision Transformer (ViT) paradigm.
- **Teacher Model:** `vit_large_patch16_384.augreg_in21k_ft_in1k`
  - A massive 300 million parameter model pretrained on ImageNet-21k and fine-tuned on ImageNet-1k.
  - Capable of extreme feature extraction at a high 384x384 resolution.
- **Student Model:** `vit_small_patch16_384.augreg_in21k_ft_in1k`
  - A heavily parameterized but mathematically leaner model (22 million parameters).
  - Acts as the blank canvas to absorb the Teacher's knowledge.

### 2.2 Asymmetric Token Pruning (ATP)
To achieve extreme inference speedups, standard ViTs—which process all background patches equally—are inadequate. We implemented **Asymmetric Token Pruning (ATP)**.
1. **Scoring Mechanism:** At designated block depths (Layers 3, 6, and 9), the model extracts the self-attention weights generated by the `CLS` (Class) token. Because the `CLS` token aggregates global image context, its attention weights naturally highlight "foreground" patches (e.g., the dog) and ignore "background" patches (e.g., the sky).
2. **Top-K Hard Dropping:** We sort these attention weights and physically drop the bottom-scoring tokens. The tensor dynamically shrinks in the sequence dimension $N \rightarrow k$.
3. **Progressive Pruning Schedule:** Token dropping is violent to a network's gradients. To prevent model collapse, we implemented a scheduled curriculum:
   - **Warmup:** Epochs 1-2 (0% dropping, allowing the model to stabilize its weights).
   - **Ramp:** Epochs 3-6 (linear interpolation dropping from 0% to the target severity).
4. **Target Keep Ratios:** We configured the per-stage keep ratios to `[0.75, 0.50, 0.40]`. Because each stage prunes the tokens that survived the previous stage, the ratios **compound**: of the original 576 patch tokens, ~432 remain after Layer 3 (75%), ~216 after Layer 6 (37.5%), and ~87 after Layer 9 (**~15% of the original tokens**). The final, most expensive Transformer blocks therefore operate on roughly one-seventh of the patches, drastically reducing their FLOPs.

## 3. Comprehensive Training Pipeline & Hyperparameters

### 3.1 Data Preparation
- **Dataset:** CIFAR-10 (50,000 train, 10,000 test).
- **Resolution:** Native 32x32 images were aggressively upscaled to **384x384** using bicubic interpolation to match the pre-trained positional embeddings of the ViT models.
- **Augmentation:** Aggressive **MixUp** (`alpha=0.8`) and **CutMix** (`alpha=1.0`) were utilized. This blends images together (e.g., 60% Cat, 40% Truck), acting as an extreme regularizer to prevent memorization.

### 3.2 Optimization & Hyperparameters
- **Optimizer:** `AdamW` with `weight_decay=0.05`.
- **Learning Rate:** `5e-5` base LR.
- **Scheduler:** `CosineAnnealingLR` combined with a `LinearLR` warmup. The cosine curve aggressively decays the LR to `1e-6` at the end of training, allowing the model to explore shallow local minima early on before sharply settling into a deep minimum (the late-stage convergence spike).
- **Batch Size:** 128 (Maximize A100 Tensor Core saturation).
- **Precision:** `torch.amp.autocast` (FP16 Automatic Mixed Precision) to accelerate matrix multiplications.

### 3.3 Distillation Configuration
- **Loss Function:** KL-Divergence balancing the hard Cross-Entropy labels and the soft KD probabilities.
- **Distill Alpha:** `0.8` (weights the Teacher's soft targets 80% and the hard ground-truth labels 20%).
- **Distill Temperature:** `T=2.0`. This scales down the raw logits before softmax, softening the probability distribution to reveal the Teacher's "dark knowledge" (e.g., teaching the student that a dog is 90% dog, but mathematically shares 8% geometric similarity with a cat).

## 4. Engineering Roadblocks & Novel Solutions

### 4.1 Roadblock 1: AToM (Attention Token Absorption) Overhead
**The Problem:** Our initial approach utilized Token Absorption (AToM), which merges background tokens into a single summary token instead of deleting them. While theoretically superior in preserving information, PyTorch's scatter/gather operations required to route and merge these tokens were heavily bottlenecked by GPU memory bandwidth. It yielded zero wall-clock speedup.
**The Solution:** We entirely stripped out AToM and pivoted to **ATP (Asymmetric Token Pruning)**. By performing a simple `torch.topk` sort and physically slicing the query tensor to the Top-K tokens (while keeping K/V at full length for a full-context read), we obtained a real wall-clock speedup at server batch sizes (~1.4× at batch 32–128) with minimal accuracy loss. Note this speedup is *batch-size dependent* — at batch size 1 the pruner overhead actually makes the model slower (§5.3).

### 4.2 Roadblock 2: The 35-Hour KD Compute Bottleneck
**The Problem:** Running standard Knowledge Distillation requires passing every augmented training batch through the 300M parameter Teacher model on the fly. This crippled the GPU, reducing training speed to 6.4 seconds per iteration and ballooning the estimated training time to **35 hours**.
**The Solution: Offline Logit Caching with MixUp Linearity**
- **Step 1:** We isolated the Teacher, passed the 50,000 clean CIFAR-10 images through it once, and saved its output probabilities (a `[50000, 10]` FP32 tensor) directly to disk (a tiny 2MB file).
- **Step 2:** MixUp normally renders offline caching impossible because you cannot pre-calculate the infinite random blends of augmented images. To bypass this, we wrote a custom `KDMixup` module that exploits the mathematical linearity of MixUp. 
- **Step 3:** During training, when the script randomly blends a Cat and a Truck at a 60/40 ratio, our script intercepts the pre-cached *clean* logits from disk and mathematically blends the logits at that exact same 60/40 ratio.
**The Result:** We perfectly mathematically simulated the Teacher without actually running the Teacher, dropping training time from 35 hours down to **2.5 hours (a ~14x speedup)**.

## 5. Final Results & Hardware Metrics

### 5.1 Accuracy Milestones
- **Teacher Accuracy (`vit_large_patch16_384`, dense):** **99.34%** on CIFAR-10 after 50 epochs.
- **Dense reference (`vit_base_patch16_224`):** **99.16%**.
- **Student Accuracy (`vit_small_patch16_384`, pruned):** **98.92%** test accuracy (measured on the full test set from `best.pt`; final-epoch CSV value 98.85%).
- **Same student with pruning disabled (keep=1.0):** **99.00%** — i.e. token pruning costs only **~0.08 pt** of accuracy.

*Analysis: Retaining 98.9% while the deep blocks operate on only ~15% of the original tokens (≈85% dropped, compounded across three stages) demonstrates successful knowledge transfer and the robustness of the ATP algorithm. The pruning-on vs pruning-off ablation (98.92% vs 99.00%) is the cleanest evidence that the accuracy cost is negligible.*

### 5.2 Total Compute Time (A100 80GB GPU)
- **Teacher Training (50 Epochs):** ~30 Minutes
- **Teacher Logit Caching (50k images):** ~2 Minutes
- **Student Training (100 Epochs):** ~2.5 Hours (Speed: ~0.38s/iteration)
- **Total End-to-End Project Compute Time:** **~3 Hours** *(Down from the original ~35+ hours)*.

### 5.3 Inference Efficiency (Measured, `vit_small_patch16_384`, A100 80GB)

Same backbone throughout. The dense numbers use a clean ViT-Small (no pruner modules, flash-attention intact); the pruned numbers use the trained ATP checkpoint with keep ratios `0.75/0.5/0.4`. Generated by `scripts/benchmark_dense_vs_pruned.py`; raw data in `outputs/benchmark_results.json`.

| Metric | Dense | Pruned (ATP) | Change |
|---|---|---|---|
| Test accuracy | 99.00% | 98.92% | −0.08 pt |
| Final-stage tokens | 576 | 87 | ~15% kept (≈85% dropped) |
| Parameters | 21.82 M | 23.59 M | +8.1% (pruner modules) |
| GFLOPs (bs=1) | 12.45 | 9.32 | **−25.1%** |
| Throughput @ bs=1 (img/s) | 145.4 | 90.7 | **0.62× (slower)** |
| Throughput @ bs=32 (img/s) | 307.8 | 430.7 | **1.40×** |
| Throughput @ bs=128 (img/s) | 328.9 | 445.6 | **1.35×** |
| Peak VRAM @ bs=128 (MB) | 1972 | 3383 | +71.6% (more) |

**Honest reading of these numbers:**
- **Accuracy is essentially preserved** (−0.08 pt) while FLOPs drop ~25% — the core DynamicViT claim reproduced.
- **The wall-clock speedup is real but batch-size-dependent:** ~1.35–1.40× at server batch sizes (32, 128), but **slower at batch size 1**, where the pruner overhead (Top-K, gather, and disabling fused/flash attention at the three pruned layers to expose attention weights) outweighs the compute saved.
- **VRAM increases, not decreases:** the *asymmetric* design keeps full-length K/V at the pruned layers (to read the whole background), trading memory for a full-context read. A pure token-dropping variant (DynamicViT-style) would instead save memory. This is a genuine limitation of the ATP-Lite design and a good discussion point.
